When Agents
Misbehave

Trust in agents is built in the moments they fail, not in the ones they succeed.

Eight design patterns

Most of the design effort today goes into the happy path: the well-prompted task, the accurate response, the smooth handoff between model and tool. Teams prototype the ideal interaction, test the intended flow, and refine the moment where everything goes right.

Everything past the happy path goes largely undesigned. It is a second priority when teams build agentic experiences.

What happens when an agent misreads your intent and executes confidently in the wrong direction? When it runs out of context mid-task and goes silent? When it hallucinates so fluently that a junior developer ships the result without noticing? These moments are where agent experiences live, and they remain largely undesigned.

To understand what good looks like in those moments, we spoke with twelve practitioners who build agentic systems for a living, or use them daily as part of their work — engineers, researchers, product managers, and designers. Between them they have shipped agents for supply chain decisioning, model customization, code generation, and enterprise data analysis. The findings here are not a survey. They are a field report.

In earlier work we described coordination as the central design problem for agentic systems: the interplay between human involvement, AI salience, and what the agent is doing behind the scenes. Misbehavior is what happens when that coordination breaks down. The principles that follow are design responses to the specific ways it breaks.

A taxonomy of misbehavior

Before getting to principles, it helps to name the categories. Across the conversations, five types of agent failure came up again and again. Each represents a different kind of breakdown in the contract between human and agent.

Wrong intent is when the agent misreads what you wanted and executes in the wrong direction. Often there is no signal it is happening. The agent proceeds confidently. You only notice when it surfaces something that does not fit, or when you read the output carefully enough to catch the drift.

Limited ability is when the agent reaches a ceiling it cannot cross: a context window running out, a tool it cannot access, a task genuinely outside its capability. The problem is not the ceiling. The problem is that the ceiling is usually invisible to the user until it has already been hit.

Hallucination is the most notorious failure mode. The agent produces output that is plausible, confident and wrong. Unlike the others, this one does not announce itself. It requires the user to already know enough to catch it, which means users who do not have that knowledge are the most exposed.

No answer is when the agent encounters a missing resource, a permissions wall, or a genuine absence of information and either goes silent or produces a response so generic it amounts to nothing. The failure is not the absence of an answer. It is the absence of a useful next step.

Context disruption is what happens when a user changes direction mid-task: redirecting, interrupting, shifting scope. Agents handle this badly far more often than they handle it well. Most absorb the new input and continue, with no acknowledgement of what changed or what was left behind.

These five categories are not exhaustive. They cover the ground where trust is most commonly damaged, and where careful design can recover it.

How each failure actually plays out

On wrong intent: the cost of silent confidence

Most agents give no signal when they have misunderstood. They proceed. The user only finds out by reading carefully, noticing the output has drifted, or getting to the end of a long execution and finding the result wrong.

One practitioner described a three-stage review process he had built for himself: he reads the chat interface as the agent writes, then checks the code, then reviews again when raising a change request. Three manual checkpoints to compensate for a design gap: no confirmation that the agent had understood the task before acting on it.

The fix participants wanted was simple and almost universally consistent: a brief, optional intent summary before execution begins. Not a confirmation dialog. Not a blocker. A short-form restatement that gives the user a chance to catch a misread before it cascades.

Even if my prompt was four sentences long, it says: OK, I understand you want to replace your fireplace, here are the things I found. There is a confirmation, then an approach.

The other insight here is about mode. Wrong-intent failures cluster in fast or agentic modes, where the agent is optimized for throughput rather than deliberation. Slower, spec-driven modes produce fewer misfires. The design implication: the faster the mode, the more important the intent check becomes.

On limited ability: design the ceiling, not the floor

Context windows run out. APIs hit rate limits. Tools fail silently. Access permissions block execution. These are not edge cases. They are regular events in any sustained agent workflow. Almost none of them are designed for.

The most striking contrast that emerged was between Claude CLI and Kiro (a coding agent many participants used at work). Claude CLI shows a running indicator of remaining context. You see when you are at 15%, plan your next prompt accordingly, ask for a summary before starting a new session. Kiro does not. It throttles silently, degrades performance without explanation and leaves users wondering whether it is broken or slow.

When I work in Claude CLI I'm a very performant developer. In Kiro I constantly find myself hitting that limit without knowing, having to retrain it from scratch.

This is a solvable problem. The pattern is well understood: surface approaching limits before they become crises. A progress indicator, a quiet warning, a gentle prompt. Any of these gives the user agency over what happens next. The current norm gives them nothing until it is too late.

When an agent cannot do something, participants consistently wanted two things in sequence: an honest explanation of what failed and why, followed immediately by a concrete recommendation for what to do instead. Error without direction is noise. Direction without explanation does not build understanding. Both together is a guided recovery.

On hallucination: the expertise gap is a design problem

Hallucination is widely discussed. What is less discussed is how differently people experience it depending on their level of domain expertise.

Senior engineers in the study spotted hallucinations quickly, often within seconds of reading output, because they already knew what correct output should look like. They would catch an agent writing mock data when real APIs existed, or implementing a frontend pattern that no experienced developer would use, because they had the reference model to compare against.

Junior developers often did not catch it. In several cases described, they approved and shipped agent-generated code that senior engineers later identified as structurally broken. The same agent, the same failure mode. Radically different outcomes depending on who was reviewing.

This is not a model problem. It is a design problem. Most agent interfaces treat hallucination as something to be explained in documentation rather than something to be surfaced in the product. Users are expected to bring their own skepticism. The users who most need protection are least equipped to provide it.

If my junior engineer has a crazy idea, they'll come to me and say, 'I have this idea, is it worth pursuing before I spend time on it?' That's what I want from the agent.

The deeper finding is worth sitting with: agents are currently trained such that saying "I don't know" is penalised equivalently to giving a wrong answer. This is why they guess, confidently, fluently, incorrectly, rather than admitting uncertainty. The design of the failure state is downstream of a training objective that treats honesty as a weakness. Surfacing uncertainty explicitly, marking low-confidence outputs visually and building in friction before high-stakes actions are all levers that matter.

On no answer: silence is the worst response

When an agent has no answer, no access, no data, no resource, no path forward, the worst thing it is able to do is nothing. An abrupt stop, a blank response, a generic error message that gives no guidance on what to try next. This was described as one of the most frustrating failure states across the board.

The pattern participants wanted was not complicated: acknowledge what failed, explain why if possible, offer a concrete alternative or next step. Even something like "I cannot find what you are looking for, but here are the things that do exist in this space" is dramatically better than silence. Proactive suggestion transforms a dead end into a redirect.

I don't want the agent to stop. I want it to say: try again, or here's what you could do instead.

The emotional register matters too. An agent that goes silent feels broken. One that acknowledges the gap and pivots feels like a collaborator.

On context disruption: acknowledge the pivot, do not absorb it

When users interrupt a running task, changing direction, adding context, redirecting, most agents absorb the new instruction and continue without acknowledging what changed. No confirmation of the new direction. No summary of what was in progress. No clean break.

The result is a kind of context soup: the agent is ostensibly following new instructions while still carrying the weight of the old ones. Participants described opening fresh conversation windows as a workaround, not because it was a good solution, but because it was the only way to ensure the old context was gone.

Two behaviors participants called out as especially problematic: the agent does not acknowledge that an interruption happened at all (silently incorporates it), and the agent over-apologises when corrected. Both erode trust, in different directions. The first suggests the agent is not tracking. The second is performative rather than functional.

What people want after an interruption: a brief, factual confirmation of the new direction and then movement. "Got it, switching to X" and then doing X. No drama, no excessive apology, no re-executing everything that already happened. Acknowledge and act.

Three tensions that cut across everything

Before stating principles, it is worth naming the underlying tensions. These are the design problems the principles respond to.

Transparency vs. speed

Every participant wanted more visibility into what the agent was doing and why. Most of them also wanted things to be fast. These goals exist in real tension. The resolution is not to pick one. It is to let transparency be modulated by trust. Early in a working relationship, show more. As patterns stabilise and trust builds, abstract progressively.

The expertise gradient

The most important finding in the research may be the simplest: the same agent failure has completely different consequences depending on who is experiencing it. Senior engineers catch hallucinations and course-correct. Junior developers miss them and ship. Agents designed for expert users, with no guardrails, no uncertainty signalling, no scaffolding, are not complete products.

Confidence vs. honesty

There is a trained incentive in most current models to complete rather than to admit. To answer rather than to say "I don't know." This produces confident, fluent, wrong output. The design response is to create space for honesty. To make uncertainty visible, to reward "I'm not sure" over a hallucinated answer, to treat an agent that flags its own limitations as a feature.

Eight principles for designing agent misbehavior

These principles are grounded in what practitioners described wanting, not what they described getting. Each is paired with a design pattern that shows what the principle looks like in practice.

01

Principle

Confirm before you commit

Before acting, the agent briefly surfaces its understanding of intent, creating a lightweight coordination moment that keeps human involvement appropriately high at initiation, when misreads are most costly.

Especially in fast or agentic modes, give the user a brief, optional restatement of how you have understood their intent before taking action. Not a dialog box. Not a blocker. A sentence. The user can override it. Most of the time they will not need to. The moments when they would have needed to are exactly the moments that currently cascade into long, frustrated recovery sessions.

Pattern — intent checkpoint

V
Analyze the Q3 sales data and summarize trends by region
Before I proceed I'll analyze the Q3 sales CSV, calculate regional totals and growth rates, and return a written summary with key trends. I won't modify any files.
Design note The card is shown for ~4 seconds then auto-proceeds. It is a checkpoint, not a gate. It anchors the coordination zone at initiation: human involvement is briefly high, then relaxes as the agent acts. The "Adjust" path opens an inline text field. Position: between the user message and the agent's first action step, not as a modal.
02

Principle

Fail transparently, recover with direction

When AI activity hits a wall, the agent makes that visible rather than silent, and immediately offers a path forward. Transparency about what is happening behind the scenes is the minimum condition for maintaining trust.

Every failure state should do two things: explain what happened and offer a concrete path forward. "I cannot access that resource" is not enough. "I cannot access that resource, here are the files I can see that might help" is designed. Raw error codes serve engineers debugging a system. They do not serve users trying to get something done. The tone should be matter-of-fact. The next step should be specific.

Pattern — structured error + recovery

V
Pull last month's model training metrics from the dashboard
✕
Couldn't access the dashboard

Your session token for the metrics dashboard has expired. I can't retrieve live data without re-authentication.

What you can do

Re-authenticate and retry automatically
Use the exported CSV from last week's report instead
Paste the metrics directly and I'll analyze them
Design note The failure message is plain language, not an error code. The three recovery paths are ranked by least friction first. Each one re-establishes a coordination zone the user can enter. The agent does not apologise. Surfacing AI activity at the point of failure, not after, is what makes recovery feel designed rather than accidental.
03

Principle

Signal uncertainty before the rabbit hole

When task complexity is high or user intent is ambiguous, the agent raises its own salience before going deep, pulling human involvement back up before it is too late to course-correct.

When confidence is low, because the prompt is ambiguous, the domain is unfamiliar, the task is genuinely novel, say so before going deep, not after. The junior engineer analogy was cited repeatedly: a good junior flags uncertainty before spending two days on the wrong thing. The equivalent agent behavior is to surface low confidence early, invite clarification and only proceed into extended execution once there is reasonable alignment.

Pattern — pre-execution uncertainty checkpoint

V
Fix the latency issue in the model serving pipeline
Confidence
42%

I can see 3 possible causes for this latency issue, but I'm not confident which one applies to your setup without more context. If I guess wrong I could take this in the wrong direction for a while.

Before I dig in — which environment is affected?
Design note The confidence meter is shown numerically and as a bar. Explicit, not decorative. This is responsive salience in action: the agent increases its own visibility when perceived risk is high, rather than waiting for the user to ask. The checkpoint is triggered only below a configurable threshold (e.g. <60%). The "skip" escape hatch ensures it never becomes a blocker.
04

Principle

Honesty over performance

Because agentic AI is nondeterministic, confident-sounding output can still be wrong. Tiered output makes that legible. Users see not what the agent produced, but how much the agent knows.

"I don't know" is better than a hallucinated answer. This requires a design stance, not a training fix, because the impulse to complete rather than to admit runs deep in how current models are shaped. Design that creates space for honesty includes: marking uncertain output visually, adding friction before high-stakes actions, giving users easy mechanisms to flag outputs that do not seem right. The agent that admits its limits is more trustworthy than the one that papers over them.

Pattern — tiered confidence output

V
What caused the spike in inference errors last Tuesday?
Confirmed from logs

The error spike began at 14:32 UTC. It coincided with a deployment of model version 3.1.2 to the eu-west-1 cluster. Rollback was initiated at 15:08 UTC and errors dropped to baseline within 6 minutes.

Inferred — not confirmed

The most likely root cause is a memory allocation change in v3.1.2 that interacts poorly with the current GPU driver version on eu-west-1 instances. I don't have access to the deployment diff to verify this directly.

Design note Two tiers only: confirmed and inferred. The visual separation is color + badge, not a footnote asterisk. Nondeterministic systems require users to calibrate their own trust. This pattern makes that calibration possible. Users flag inaccurate sections inline, feeding back into the agent's context. Never three or more tiers. It becomes noise.
05

Principle

Build a salience arc, not a fixed mode

AI salience, how prominent and active the agent feels, should shift over time. High salience means the agent is visible, collaborative and explains itself. Low salience means it works quietly and autonomously. The right position changes as trust builds.

Agents should not interact the same way on day one as they do on day thirty. Early interactions benefit from more explicit confirmation, slower pacing, more visible reasoning. As trust accumulates, as the user's mental model solidifies and the agent's understanding deepens, the interaction should progressively abstract. Show less, confirm less, move faster. Autonomy is earned, not assumed.

Pattern — responsive salience control

Design note Salience is driven by observable coordination signals: task complexity, perceived risk, correction frequency, not elapsed time. High salience increases human involvement and makes AI activity visible. Low salience steps back and lets the agent work. The current salience level is always shown so users are never surprised by how much the agent is doing on their behalf. Users override at any moment.
06

Principle

Design for the skill gradient

User comfort level and domain expertise are coordination signals. The same failure state should produce different salience and different amounts of AI activity depending on who is experiencing it.

The users who most need protection from agent failures are the ones least equipped to protect themselves. This is not a reason to dumb down the experience. It is a reason to make error states, uncertainty signals and confirmation patterns adaptive. What a senior engineer needs at a moment of failure is different from what a first-year developer needs. The same failure state should not produce the same interface response for both.

"Developer" shows structured technical data for users who want more AI activity visible. "Raw logs" for engineers who need full transparency. The default tab is set per user role, not locked globally. This is coordination calibrated to who is in the loop.
07

Principle

Announce limits before they bite

Coordination curves have valleys: stretches where the agent operates independently and human involvement drops. When those valleys approach a hard limit, salience should rise enough to let the user plan their next move.

Context windows, rate limits, tool access, knowledge cutoffs: these are features, not secrets. A well-designed agent surfaces approaching limits in time for users to do something about them. The pattern from Claude CLI, showing remaining context percentage as you work, was cited as one of the most useful behaviors any agent had produced. It changes the user's behavior in the moment. That is what good signalling does.

The message is always actionable: not "low context" but "consider summarizing before your next task."
08

Principle

Acknowledge and act, don't apologise

Context disruption is a coordination event. The agent should handle the handoff cleanly: marking what was in progress, confirming the new direction and restoring momentum, without treating it as an emotional moment.

When corrected or interrupted, an agent should acknowledge the new direction and move on. Not apologise three times, not re-explain everything it had been doing, not perform contrition. "Got it, adjusting to X" and then adjusting to X. Over-apologising was described as actively frustrating. It adds latency without adding value and makes the agent feel fragile rather than capable. The goal after an interruption is momentum, not emotional management.

Pattern — clean pivot acknowledgement

A
I'm refactoring the authentication module — renaming the token handlers and restructuring the session middleware…
V
Stop — actually I need to fix the login bug first before we refactor anything
Paused Auth module refactor — 30% complete
↩ Switching to login bug. Refactor paused and saved — you can resume it later.
A
OK — what's the login bug? Can you describe what's failing, or point me to the relevant file?
Design note Three elements, in sequence: (1) the paused task is shown struck-through and labelled "Paused," the coordination curve dips but the thread is preserved; (2) a single dark toast acknowledges the pivot and confirms continuity; (3) the agent immediately moves human involvement back up by asking for what it needs. No apology. No recap. A good handoff looks like this: acknowledge, confirm, proceed.

What designing for misbehavior means

Designing for misbehavior is not the same as designing for failure. It is designing for trust.

An agent that handles errors gracefully, signals uncertainty honestly, confirms before it acts and pivots cleanly when redirected is not a lesser agent than one that always succeeds. It is a more trustworthy one. Trust does not come from never failing. It comes from how failure is handled.

The happy path is not where users form their lasting impressions of an agent. The moment where something goes wrong, and the agent either handles it well or does not, is where character is revealed. Right now, almost no one is designing that moment.

They should be.

This piece draws on research conducted across twelve practitioners building and using agentic AI systems in enterprise contexts. Participants spanned engineering, product, design and research roles.
