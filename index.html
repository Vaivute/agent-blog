Salience · Coordination · Trust
01

Principle

Confirm before you commit

Before acting, the agent briefly surfaces its understanding of intent — creating a lightweight coordination moment that keeps human involvement appropriately high at initiation, when misreads are most costly.

Pattern — intent checkpoint

V
Analyse the Q3 sales data and summarise trends by region
Before I proceed I'll analyse the Q3 sales CSV, calculate regional totals and growth rates, and return a written summary with key trends. I won't modify any files.
Design note The card is shown for ~4 seconds then auto-proceeds — it's a checkpoint, not a gate. It anchors the coordination zone at initiation: human involvement is briefly high, then relaxes as the agent acts. The "Adjust" path opens an inline text field. Position: between the user message and the agent's first action step, not as a modal.
02

Principle

Fail transparently, recover with direction

When AI activity hits a wall, the agent makes that visible rather than silent — and immediately offers a path forward. Transparency about what's happening behind the scenes is the minimum condition for maintaining trust.

Pattern — structured error + recovery

V
Pull last month's model training metrics from the dashboard
✕
Couldn't access the dashboard

Your session token for the metrics dashboard has expired. I can't retrieve live data without re-authentication.

What you can do

🔑 Re-authenticate and retry automatically
📂 Use the exported CSV from last week's report instead
✏️ Paste the metrics directly and I'll analyse them
Design note The failure message is plain language, not an error code. The three recovery paths are ranked by least friction first — each one re-establishes a coordination zone the user can actually enter. The agent does not apologise. Surfacing AI activity at the point of failure, not after, is what makes recovery feel designed rather than accidental.
03

Principle

Signal uncertainty before the rabbit hole

When task complexity is high or user intent is ambiguous, the agent raises its own salience before going deep — pulling human involvement back up before it's too late to course-correct.

Pattern — pre-execution uncertainty checkpoint

V
Fix the latency issue in the model serving pipeline
Confidence
42%

I can see 3 possible causes for this latency issue, but I'm not confident which one applies to your setup without more context. If I guess wrong I could take this in the wrong direction for a while.

Before I dig in — which environment is affected?
Design note The confidence meter is shown numerically and as a bar — explicit, not decorative. This is responsive salience in action: the agent increases its own visibility when perceived risk is high, rather than waiting for the user to ask. The checkpoint is triggered only below a configurable threshold (e.g. <60%). The "skip" escape hatch ensures it never becomes a blocker.
04

Principle

Honesty over performance

Because agentic AI is nondeterministic, confident-sounding output can still be wrong. Tiered output makes that legible — users see not just what the agent produced, but how much the agent actually knows.

Pattern — tiered confidence output

V
What caused the spike in inference errors last Tuesday?
Confirmed from logs

The error spike began at 14:32 UTC. It coincided with a deployment of model version 3.1.2 to the eu-west-1 cluster. Rollback was initiated at 15:08 UTC and errors dropped to baseline within 6 minutes.

Inferred — not confirmed

The most likely root cause is a memory allocation change in v3.1.2 that interacts poorly with the current GPU driver version on eu-west-1 instances. I don't have access to the deployment diff to verify this directly.

Design note Two tiers only — confirmed and inferred. The visual separation is colour + badge, not just a footnote asterisk. Nondeterministic systems require users to calibrate their own trust; this pattern makes that calibration possible. Users can flag inaccurate sections inline, feeding back into the agent's context. Never three or more tiers — it becomes noise.
05

Principle

Build a salience arc, not a fixed mode

AI salience — how prominent and active the agent feels — should shift over time. High salience means the agent is visible, collaborative, and explains itself. Low salience means it works quietly and autonomously. The right position changes as trust builds.

Pattern — responsive salience control

Salience Session 14 · 3 weeks active
Low salience ↑ Now
Calibrated High salience
Low salience
Done-for-me · agent acts autonomously, surfaces only blockers, summarises after
Calibrated
Done-with-me on complex steps · done-for-me on routine ones · adapts per task
High salience
Done-with-me · agent confirms intent, narrates reasoning, checks before acting
Salience shifts automatically based on task complexity, perceived risk & user comfort · override manually at any time
Design note Salience is driven by observable coordination signals — task complexity, perceived risk, correction frequency — not just elapsed time. High salience increases human involvement and makes AI activity visible; low salience steps back and lets the agent work. The current salience level is always shown so users are never surprised by how much the agent is doing on their behalf. Users can override at any moment.
06

Principle

Design for the skill gradient

User comfort level and domain expertise are coordination signals just as much as task complexity and perceived risk. The same failure state should produce different salience and different amounts of AI activity depending on who's experiencing it.

Pattern — layered error disclosure

Simple
Developer
Raw logs
⚠️
The training job couldn't start The system doesn't have enough GPU capacity right now. This usually clears up within 30–60 minutes. You can wait and retry, or switch to a smaller instance type to start sooner.
Technical details ▾
ResourceLimitExceeded: ml.p3.8xlarge capacity
Region: us-east-1 · Requested: 4 instances
Available: 0 · Queue position: 12
Estimated wait: 45–90 min

Alternative: ml.p3.2xlarge available (×8 now)
Estimated training delta: +2.3h
Design note Three tabs, not one. The "Simple" tab is what everyone sees by default — written for a user whose comfort level with the domain is low. "Developer" shows structured technical data for users who want more AI activity visible. "Raw logs" for engineers who need full transparency. The default tab is set per user role, not locked globally. This is coordination calibrated to who's in the loop.
07

Principle

Announce limits before they bite

Coordination curves have valleys — stretches where the agent operates independently and human involvement drops. When those valleys are approaching a hard limit, salience should rise just enough to let the user plan their next move.

Pattern — progressive limit indicator

18% context remaining — consider summarising before starting a new task
18%
A
Here's the refactored authentication module. I've separated the OAuth flow from the session management and added proper error boundaries throughout…
Comfortable
80–100%
No indicator shown
Approaching
15–79%
Bar appears, no action needed yet
Critical
<15%
Prompt to summarise & continue
Design note The bar only appears when the limit is worth being aware of (<80%). At comfortable capacity there's no visual noise — salience stays low. As the limit approaches, salience rises just enough to prompt action. The colour transitions warm as limits close in — amber, not red — because this is a coordination signal, not an alarm. The message is always actionable: not "low context" but "consider summarising before your next task."
08

Principle

Acknowledge and act — don't apologise

Context disruption is a coordination event. The agent should handle the handoff cleanly — marking what was in progress, confirming the new direction, and restoring momentum — without treating it as an emotional moment.

Pattern — clean pivot acknowledgement

A
I'm refactoring the authentication module — renaming the token handlers and restructuring the session middleware…
V
Stop — actually I need to fix the login bug first before we refactor anything
Paused Auth module refactor — 30% complete
↩ Switching to login bug. Refactor paused and saved — you can resume it later.
A
OK — what's the login bug? Can you describe what's failing, or point me to the relevant file?
Design note Three elements, in sequence: (1) the paused task is shown struck-through and labelled "Paused" — the coordination curve dips but the thread is preserved; (2) a single dark toast acknowledges the pivot and confirms continuity; (3) the agent immediately moves human involvement back up by asking for what it needs. No apology. No recap. A good handoff looks like this: acknowledge, confirm, proceed.
